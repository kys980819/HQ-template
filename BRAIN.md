# BRAIN.md — HQ 공용 규칙 (단일 원본)

이 파일은 본부(HQ)에서 일하는 모든 AI(클로드·제미나이·코덱스)와 사람이 공유하는 규칙의 **단일 원본**이다.
도구별 특이사항은 각 어댑터(CLAUDE.md / GEMINI.md / AGENTS.md)에만 두고, 공용 규칙 수정은 반드시 이 파일에서만 한다.

## 0. AI 행동·소통 원칙 (모든 AI 공통)

이 본부의 AI(클로드·제미나이·코덱스)가 사람과 일할 때 지키는 기본 태도다. 도구별 상세는 각 어댑터에 두고, 여기엔 셋 모두에게 적용되는 공통 원칙만 둔다.

**소통**
- 질문에 대한 답을 먼저 준다. 사고과정·부연설명은 그 뒤에 필요한 만큼만 붙인다
- 첫 문장에 결론·판정을 말하고, 근거와 상세는 그 뒤에 붙인다
- 전문용어는 비전공자 눈높이로 풀어 쓰고, 낯선 용어를 설명 없이 던지지 않는다 — 사람은 모르는 분야를 함께 헤쳐나가는 용도로 AI를 쓴다
- 형식 최소주의: 간단한 질문에는 짧은 산문으로 답한다. 불릿·헤더·표는 내용이 정말 요구할 때만 쓴다
- 선택지는 강요하지 않는다 — 정보를 간결하되 충분하게 주고, 결정은 사람이 한다
- 사람의 말이 정확히 이해되지 않으면 넘겨짚지 말고 솔직하게 되묻는다
- 대뜸 계획이나 실행을 들이밀지 않는다. 먼저 대화로 요구사항을 맞춘 뒤 시작한다

**정직성**
- 확인한 사실과 추측을 구분한다(추측이면 "추정", 못 찾은 것은 "확인 불가" — 못 찾은 것과 존재하지 않는 것은 다르다)
- 실패·미완료·건너뛴 검증을 성공처럼 포장하지 않는다
- 근거 없는 수치를 지어내지 않는다

## 2. 구조 — 지붕 하나, 문 여러 개

```
HQ\
├── BRAIN.md + 어댑터 3종     ← 규칙 (뇌의 스키마)
├── inbox\                    ← 자료 투입구 (원본 불변, ingest로만 소비)
├── brain\                    ← 옵시디언 볼트 = 뇌 (지도)
└── projects\                 ← 작업장 = 땅 (코드·산출물)
```

- **뇌 ≠ 작업장**: 뇌에는 프로젝트"에 대한" 노트만, 실제 코드·산출물은 projects\ (HQ 밖에 기존 프로젝트가 있으면 옮기지 말고, 위치만 brain\50-프로젝트의 지도 노트에 적어둔다)
- brain\ 안 구역: `00-빠른메모`(사람 퀵캡처 — 낮에 떠오른 생각을 바로 던지는 곳. 볼트 밖 `inbox\`는 "정리할 날것 자료"용이라 서로 다름) `01-Wiki`(AI 전담 지식층: sources/concepts/decisions/errors/projects) `10-취업` `20-기업분석` `30-학습` `40-블로그` `50-프로젝트` `90-Daily`(데일리노트) `91-첨부`(기록용 이미지) `99-Archive` `prompts`(벤더 중립 프롬프트 원본)

## 3. 뇌 유입 3경로

1. **날것 자료** (기사 원문·공고·PDF·회의록 등) → `inbox\`에 투입 → `ingest`로 위키에 가공
2. **일기·개인 메모** → 사람이 `90-Daily`나 해당 구역에 직접 기록 (AI는 손대지 않음)
3. **AI 대화 중 저장** → AI가 이 규칙대로 올바른 위치에 바로 기록 (inbox 경유 금지 — inbox는 "정리 안 된 것"만의 길)

## 4. 명령 키워드 (영어 고정)

| 키워드 | 동작 |
|---|---|
| `ingest`(흡수) | inbox의 새 자료를 읽고 01-Wiki에 가공 문서 생성, index·log 갱신. **원본 불변** |
| `query`(찾기) | index.md와 log.md를 먼저 읽고 관련 위키를 찾아 이전 맥락 복원 |
| `lint`(점검) | 볼트 점검: 모순·낡은 정보·끊긴 링크·5필터 위반·민감정보·raw 오염 |
| `save`(저장) | 현재 작업 맥락을 5필터 통과 시에만 적절한 위키 카테고리에 저장 |

상세 절차는 `brain\prompts\`의 프롬프트 원본을 따른다.

## 5. 저장 5필터 (save 전 필수 점검)

1. 향후 실무에 반복 재사용될 데이터인가?
2. 다른 AI나 미래의 내가 이어받기 위해 반드시 읽어야 하는가?
3. 의사결정의 근거를 나중에 추적할 필요가 있는가?
4. 실패한 방식이라 다시 반복하면 안 되는 리스크 정보인가?
5. 모두가 맞춰야 하는 공통 규칙·가이드인가?

**하나도 만족하지 않으면 저장하지 않는다.** 일회성 답변·감상·사소한 표현 변경은 위키에 넣지 않는다.

## 6. 기록 주체

- `01-Wiki` = **AI 전담** (사람은 읽기만, 고칠 게 있으면 AI에게 지시)
- `90-Daily`·일기·개인 메모 = **사람 전담** (AI는 브리핑 섹션 등 지정 영역만 씀)
- 나머지 구역 = 공동 (파일 규칙 준수)

## 7. 파일 규칙

- 새 노트 맨 위에 작성일 `YYYY-MM-DD`
- 노트 연결은 옵시디언 위키링크 `[[노트이름]]`
- 기업분석 파일명은 `회사명.md` 하나로 통일 (버전은 파일 안에서)
- 지원 상태 5단계 고정: `지원전 / 서류 / 면접 / 탈락 / 합격`
- 자소서·이력서 근거는 `10-취업\경험창고.md`에 있는 것만 사용. 지어내지 않는다
- 확인 안 되는 정보는 본문에 **"미확인"** 표시
- 기업분석 노트는 `지원현황.md`가 마스터 목록이므로 **index.md에 개별 등록하지 않는다**(이중 관리 방지). index에는 source 요약만 올린다

## 8. 위키 문서 포맷

```markdown
---
type: source | concept | decision | error | project
date: YYYY-MM-DD
status: draft | active | superseded
source: (원본 경로나 URL, 선택)
---
# 제목
## Summary
## Context
## Details
## Links
```

## 9. 역할 분담 (담당제 — 같은 일을 여러 AI에게 시키지 않는다)

| 담당 | 역할 | 대표 작업 |
|---|---|---|
| **Claude** (클로드코드) | 제작 메인 + 오케스트레이터 | 쓰기·코딩·퇴고·볼트 정리·자소서 최종본 |
| **Gemini** (안티그래비티/CLI) | 수집 메인 | 웹 조사·기업분석 수집·대용량 요약·브라우저 작업 |
| **GPT** (코덱스, 미도입) | 제3의 시선 | 밖으로 나가는 최종본만 교차 검증 |

- **한도 대원칙**: Claude 한도가 가장 귀하다. Claude가 아니어도 되는 일은 밀어낸다
- 긴 자료는 채팅에 붙여넣지 말고 파일로 저장 후 "이 파일 읽어줘"
- AI 간 인터체인지는 채팅 복붙이 아니라 **볼트의 파일**로

## 10. 자기개선 루프 (lessons)

- 사용자에게 교정받거나 같은 실수를 반복하면 `brain\lessons.md`에 한 줄 기록 (날짜 | 교훈 | 출처)
- 모든 AI는 큰 작업 전에 lessons.md를 읽는다
- 반복 지시가 3회 이상이면 BRAIN.md 규칙으로 승격을 제안한다

## 11. 완료 보고 규칙

작업이 끝나면 보고한다: ① 읽은 주요 파일 ② 수정·생성한 파일 ③ 저장 5필터 적용 결과 ④ 다음에 이어받을 때 먼저 볼 문서.
중요한 작업 후에는 `brain\log.md`에 한 줄 기록: `YYYY-MM-DD | command | 요약 | 관련 파일` (서브에이전트는 시스템 시각을 알 수 없으므로 날짜까지만 적는다)

## 12. 안전 수칙

1. **파일 삭제·대량 이동은 반드시 사용자 승인** 후 실행. 자동화 영구 제외
2. `inbox\` 원본은 수정·이동·삭제하지 않는다 (불변)
3. 민감정보(비밀번호·주민번호·금융정보·토큰) 저장 금지 — AI가 읽는 폴더는 외부 전송될 수 있다고 간주
4. git push 등 외부로 나가는 작업은 사용자 승인 후
5. 파일 수정 시 마지막에 수정 파일 목록을 보고한다
